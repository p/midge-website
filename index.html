Midge

What is Midge?

Midge is a system for tracking bugs found during the commercial development of a software product. It is particularly suited to a process for which the filing, managing, fixing, and testing are all undertaken by different roles in a trusted environment.

Midge aims to be consistent, self-explanatory, powerful enough to efficiently manage thousands of bugs, require no administration, and be bug-free!

If you have any ideas, requests, or problems regarding Midge, please send feedback.

Principles and features

Key principles behind midge include:

  • Minimise the number of states so that the user does not accidentally miss bugs.
  • Avoid rarely used state values (e.g. use "new" again rather than "reopened")
  • Keep states as orthogonal as possible. (e.g. don't have a "resolved" category which is only relevant when the status is "closed" or "fixed".
  • Allow some states to have values dynamically decided by the user (and automatically cleaned up when unused).

Some key programming features of Midge include:

  • Developed entirely in Python.
  • Ever expanding unit test coverage.
  • Normalised SQL tables.
  • Multiple UIs (currently supports standalone http server).
  • No manual database setup/teardown.
  • Import/export API for migrating bugs to/from other systems.

Requirements

Midge has been developed and tested with the following software (earlier versions may work):

  • postgres-7.4.2
  • psycopg-1.1.11
  • python-2.3.3
  • egenix-mx-base-2.0.4
